Revise the backend plan for the standalone Social Command Center app.

Infrastructure:
- GitHub for source control
- Railway for deployment
- Railway PostgreSQL for database
- Railway backend API using Node.js with Express or Fastify
- Railway background worker for media processing
- Sharp for image resizing/cropping
- FFmpeg for video resizing/cropping/transcoding
- Cloudflare R2 or Backblaze B2 for low-cost S3-compatible object storage
- n8n for scheduled trigger automation
- Local AI computer for future caption/reply generation

Important:
Do not use Cloudinary or any heavy subscription-based media service.
Do not use paid media transformation services unless added later as an optional fallback.
The app should self-process media using Sharp and FFmpeg.

Goal:
Build the backend foundation that can store originals, generate platform-specific image/video versions, schedule posts, mock publish, manage comments, and prepare for official social API integrations.

Storage:
Use an S3-compatible storage abstraction so the provider can be changed later.

Supported storage providers:
- Cloudflare R2
- Backblaze B2
- Bunny Storage if implemented separately
- AWS S3 optional later

Media upload flow:
1. Frontend uploads original media to backend or receives a presigned upload URL.
2. Backend stores original file in S3-compatible storage.
3. Backend creates MediaAsset record.
4. Backend analyzes dimensions, duration, file type, and size.
5. Backend creates MediaVersion records for selected platform presets.
6. Backend creates MediaProcessingJob records.
7. Media worker processes jobs using Sharp or FFmpeg.
8. Worker uploads processed versions to storage.
9. Worker updates MediaVersion status and publicUrl.
10. App uses the specific MediaVersion for each platform post.

Media validation:
Create media-validation.service.ts.

Validate:
- original resolution
- target resolution
- aspect ratio compatibility
- crop severity
- upscaling required
- file type compatibility
- video duration placeholder
- platform media requirement

Quality labels:
- Excellent
- Good
- Needs Review
- Poor

Validation statuses:
- READY
- WARNING
- FAILED
- NEEDS_REVIEW
- PROCESSING

Rules:
- Warn if source is smaller than target.
- Warn if severe crop is required.
- Fail unsupported file types.
- Block Instagram/TikTok scheduling if required media is missing.
- Block publishing if selected media version is not READY or WARNING.
- Allow saving draft even if media is not ready.
